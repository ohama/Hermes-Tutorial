# 학습 루프

> **이 챕터를 시작하기 전에**
> - [Ch.9 스킬 시스템](../09-skills/index.md)을 먼저 완료하십시오 — 학습 루프의 핵심은 스킬 생성과 개선입니다.

## 개요

Do→Learn→Improve 사이클은 Hermes의 flagship 기능입니다. 복잡한 작업을 수행하면 에이전트가 그 워크플로우를 스킬로 저장할지 제안합니다. 사용자가 수락하면 해당 스킬은 유사한 미래 작업에 자동 주입되어 절차가 재사용됩니다. 그리고 백그라운드 Curator가 주기적으로 스킬 품질을 검토하고 개선합니다.

이 챕터에서는 스킬이 어떻게 제안되고 생성되는지 직접 관찰하고, 자가 개선 흔적을 확인하며, Curator를 수동으로 실행해 봅니다.

---

## 개념: Do→Learn→Improve 사이클

아래 다이어그램은 학습 루프의 전체 흐름을 보여줍니다.

```
┌─────────────────────────────────────────────┐
│  Do:   복잡한 다단계 작업 수행              │
│        (read_file / terminal / write_file 등│
│         여러 툴을 순차적으로 실행)          │
└───────────────────┬─────────────────────────┘
                    │ 작업 성공
                    ▼
┌─────────────────────────────────────────────┐
│  Learn: 에이전트가 워크플로우를 분석하고    │
│         스킬로 저장할지 제안                │
│         (사용자가 수락 → SKILL.md 생성)     │
└───────────────────┬─────────────────────────┘
                    │ 스킬 저장 완료
                    ▼
┌─────────────────────────────────────────────┐
│  Improve: 유사한 후속 작업에서 스킬이       │
│           자동 주입 → 절차 재실행 →        │
│           새로운 엣지 케이스 발견 시        │
│           스킬을 자동 patch                 │
└───────────────────┬─────────────────────────┘
                    │ 주기적 비활동 후
                    ▼
┌─────────────────────────────────────────────┐
│  Curator: 백그라운드 AIAgent fork가         │
│           스킬 품질 일괄 검토 및 개선       │
│           (비활동 기반 트리거)              │
└─────────────────────────────────────────────┘
```

---

## 개념: 1단계 — 스킬 자동 생성과 제안 (Learn)

에이전트가 복잡한 작업을 성공적으로 완료하면, 그 접근법을 스킬로 저장할지 **제안**합니다. 사용자가 수락하면 스킬은 `~/.hermes/skills/<category>/<name>/SKILL.md`에 기록됩니다.

> **검증 필요**: 스킬 저장 제안의 정확한 발동 조건과 메시지 문구, 수락 방법은
> 환경/버전에 따라 다릅니다. 아래 실습에서 로컬 실행 후 실제 출력을 캡처하십시오.

공식 문서에 따르면 에이전트는 복잡한 다단계 작업 이후 "이 워크플로우를 스킬로 저장할까요?" 유사한 제안을 합니다. 사용자가 응답하지 않으면 워크플로우는 저장되지 않습니다 — 반드시 수락 의사를 표시해야 합니다.

---

## 개념: 2단계 — 사용 중 자가 개선 (Improve)

스킬이 후속 작업에서 트리거되면 에이전트는 다음 과정을 거칩니다.

1. 스킬 전체 내용을 컨텍스트에 로드합니다.
2. 스킬에 정의된 절차를 실행합니다.
3. 실행 중 새로운 엣지 케이스나 더 나은 접근법을 발견하면, 에이전트가 `patch` 액션으로 스킬을 자동 업데이트할 수 있습니다.

스킬의 `patch_count` 메타데이터는 스킬이 몇 번 개선되었는지를 추적합니다. 독자는 자연어로 "이 스킬을 개선해줘"라고 요청하여 에이전트가 직접 스킬을 업데이트하도록 할 수도 있습니다.

---

## 실습: 스킬이 만들어지는 과정 관찰하기

**목표:** 복잡한 작업을 요청해 에이전트가 스킬 저장을 제안하는 순간을 관찰합니다.

### 1단계 — 다단계 작업 요청

```bash
# 검증: hermes rolling, 2026-06-10
hermes chat -q "이 프로젝트의 모든 Python 파일에서 미사용 import를 찾아 리포트를 만들어줘"
```

에이전트는 `read_file`, `terminal`, `write_file` 등 여러 툴을 순차적으로 실행합니다.

### 2단계 — 스킬 저장 제안 관찰 및 수락

작업이 완료되면 에이전트가 워크플로우를 스킬로 저장할지 제안합니다. 제안이 보이면 수락하십시오.

제안 메시지와 수락 결과는 로컬 환경에서 확인이 필요합니다.

```
# [로컬 실행 후 캡처 필요 — 스킬 저장 제안 메시지 및 스킬 파일 생성 확인]
```

> **검증 필요**: 위 코드블록 자리에 로컬 실행 후 실제 에이전트 제안 메시지와 수락 결과를
> 캡처하여 대체하십시오.

### 3단계 — 스킬 생성 확인

```bash
# 검증: hermes rolling, 2026-06-10
hermes skills list
ls ~/.hermes/skills/
```

새로 생성된 스킬이 목록에 보이면 성공입니다.

---

## 실습: 자가 개선 흔적 확인하기

스킬을 여러 번 사용한 후, 사용 횟수와 개선 횟수를 확인합니다.

```bash
# 검증: hermes rolling, 2026-06-10
cat ~/.hermes/skills/.usage.json
```

> **검증 필요**: `.usage.json`의 정확한 위치와 형식은 로컬 실행 후 확인하십시오.
> 파일이 없으면 `ls -a ~/.hermes/skills/`로 숨김 파일을 먼저 확인하십시오.

`use_count`는 스킬이 트리거된 총 횟수를, `patch_count`는 자동 개선 횟수를 나타냅니다.

## 개념: 3단계 — Curator 백그라운드 품질 검토

Curator는 백그라운드 AIAgent fork로 격리 실행되어 현재 대화에 영향을 주지 않으면서 스킬 품질을 일괄 검토하고 개선합니다.

**중요 — 비활동 기반 트리거:** Curator는 작업 횟수가 아니라 **비활동** 여부에 따라 실행됩니다. 기본 설정에서는 두 조건이 **동시에** 충족되어야 실행됩니다.

- `min_idle_hours` (기본 2시간): 마지막 활동 후 최소 2시간 경과
- `interval_hours` (기본 168시간 = 7일): 마지막 Curator 실행으로부터 7일 경과

신규 설치에서는 첫 인터벌(7일)만큼 첫 실행이 지연됩니다. 따라서 처음 사용하는 세션에서는 Curator가 자동으로 보이지 않습니다.

**설정 예시:**

```yaml
# 검증: hermes rolling, 2026-06-10
curator:
  enabled: true
  interval_hours: 168        # 7일 (기본값)
  stale_after_days: 30       # 30일 미사용 → stale 표시
  archive_after_days: 90     # 90일 미사용 → archived (복구 가능)
```

**Curator 실행 결과물 위치:**

- 리포트: `~/.hermes/logs/curator/<timestamp>/REPORT.md`
- 각 실행 전 백업: `~/.hermes/skills/.curator_backups/`
- 아카이브(복구 가능): `~/.hermes/skills/.archive/`

## 실습: Curator 수동 실행

자동 실행을 기다리지 않고 Curator를 직접 트리거할 수 있습니다.

```bash
# 검증: hermes rolling, 2026-06-10
hermes curator status           # 마지막 실행, 스킬 카운트, 최근 미사용 스킬 표시
hermes curator run              # 즉시 실행 (포그라운드)
hermes curator run --background # 백그라운드 실행
hermes curator run --dry-run    # 변경 없이 미리보기
hermes curator rollback         # 마지막 실행 전 상태로 복원
```

`hermes curator status` 출력은 실행 환경에 따라 다릅니다. 아래 자리에 로컬 출력을 캡처하십시오.

```
# [로컬 실행 후 캡처 필요 — curator status 출력]
```

> **검증 필요**: `hermes curator status` 출력 형식은 Hermes 버전에 따라 다를 수 있습니다.
> 로컬 실행 후 실제 출력을 캡처하여 위 자리를 대체하십시오.

## 심화: Honcho 외부 메모리 연동 (선택 — 외부 계정 필요)

Honcho는 honcho.dev에서 제공하는 외부 장기 메모리 서비스입니다. Hermes와 연동하면 세션 간 사용자 패턴을 더 풍부하게 기억할 수 있습니다. 단, **이 기능은 honcho.dev에서 별도 계정과 API 키를 발급받아야 사용할 수 있는 선택 심화 기능**입니다. 자기 호스팅(self-hosted) 옵션도 지원됩니다(`AUTH_USE_AUTH=false`로 인증 건너뛰기 가능).

설정은 간단합니다. `~/.hermes/.env`에 API 키를 추가하고, `config.yaml`에 `memory.provider: honcho`를 한 줄 설정한 뒤 `hermes memory setup honcho`를 실행합니다.

```yaml
# 검증: honcho.dev 계정 필요, hermes rolling, 2026-06-10
# 최소 Honcho 설정 (심화 — 계정 필요)
memory:
  provider: honcho
# ~/.hermes/.env에 추가:
# HONCHO_API_KEY=your-key-here
```

설치 명령:

```bash
# 검증: honcho.dev 계정 필요, hermes rolling, 2026-06-10
hermes memory setup honcho
```

Honcho 연동 시 `honcho_profile`, `honcho_search`, `honcho_context` 등 Honcho 전용 툴이 추가로 활성화되며, 추가 설정 키(`contextCadence`, `dialecticCadence` 등)는 공식 Honcho 연동 문서를 참고하십시오.

> **주의**: Honcho 연동은 honcho.dev에서 별도 계정과 API 키가 필요합니다.
> 무료 계정으로 시작할 수 있지만, 클라우드 서비스 의존성이 추가됩니다.
> 자기 호스팅(self-hosted) 옵션도 지원됩니다.

## 흔한 오류 / 주의

> **주의 1 — 스킬 저장 제안을 놓치지 말 것**
> 에이전트가 스킬 저장을 제안할 때 응답하지 않으면 해당 워크플로우는 저장되지 않습니다.
> 작업 완료 후 에이전트의 메시지를 주의 깊게 확인하십시오.

> **주의 2 — Curator는 즉시 실행되지 않음**
> Curator는 "15번째 작업마다"가 아닌 비활동 기반 트리거로 실행됩니다.
> 기본 설정에서는 마지막 활동 후 최소 2시간 경과 AND 7일 간격이 지나야 실행됩니다.
> `hermes curator run`으로 즉시 수동 실행할 수 있습니다.

> **주의 3 — Honcho는 외부 계정이 필요함**
> Honcho 연동은 honcho.dev에서 별도 계정과 API 키가 필요합니다.
> 무료 계정으로 시작할 수 있지만, 클라우드 서비스 의존성이 추가됩니다.
> 자기 호스팅(self-hosted) 옵션도 지원됩니다.
